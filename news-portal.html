Part 1: Design of Monolithic News Portal Infrastructure

In this project, I designed and managed a robust infrastructure for a high-traffic news portal. The system was architected to handle large spikes in reader traffic while providing a stable editorial management interface.

The architecture is based on a vertical and horizontal scaling strategy, ensuring that the system can handle 2-3 times the peak RPS (Requests Per Second) observed during the busiest hours.

1. Traffic Management & Edge Security

  • Load Balancer (Azure/Citrix): Acts as the primary entry point. It manages SSL termination to offload encryption tasks from the backend and distributes traffic to the healthy nodes in the backend pool.
  • CDN (Cloudflare/Akamai): Positioned in front of the Load Balancer to provide WAF protection and DDoS mitigation.

Operational Note: We avoided "Sticky Sessions" by managing all stateful data (like user sessions) in a distributed Redis cache, ensuring that if a backend server fails, the user’s session remains intact on another node.

2. Caching Layer (Varnish Cluster)

To reduce database load and improve response times, a high-performance caching layer was implemented.

  • Varnish Cache: Deployed on dedicated Virtual Machines (VMs) rather than containers to ensure maximum RAM performance. We achieved a ~90% Cache Hit ratio.
  • High Availability: A dual-node Varnish cluster was used in a "Full Mesh" configuration, where each Varnish node connects to every backend server.
  • Grace Mode: Enabled to ensure that if the backend becomes unavailable, the system continues to serve "stale" content from the cache, preventing a total site outage during backend failures.

3. Application Layer (Backend & Frontend)

The core logic was built using .NET for its enterprise-level stability and performance under heavy load.

  • Docker on VMs: Applications were containerized to ensure environment consistency across development and production.
  • Capacity Planning: We maintained a target CPU utilization of 40%-60%. This "buffer" is critical for news sites to survive sudden traffic spikes (breaking news events) without the servers locking up.
  • Frontend Strategy: We utilized a clean separation between the user-facing site (MVC) and the administrative tools to reduce complexity and improve maintainability.

4. Data & State Management

  • MongoDB (NoSQL): Implemented as a Replica Set (1 Primary, 2 Secondaries). Dedicated VMs were used instead of Docker to maximize disk I/O performance.
  • Election Flapping Prevention: Priority settings were fine-tuned to prevent unnecessary primary re-elections during minor network fluctuations, which would otherwise cause brief service outages.
  • Redis: Used for distributed locking (preventing editors from overwriting each other's work) and session management.
  • S3 Storage: All media assets and static files are offloaded to S3, integrated with the CDN for global delivery.

Pros & Cons of This Architecture

Pros:

  • Fast Time-to-Market: The monolithic structure allowed for rapid development and deployment in the early stages.
  • High Performance: Thanks to Varnish and .NET, the system served millions of requests with sub-second latency.
  • Stability: A well-understood stack that is easy to monitor using Application Insights.

Cons (The Drive to Microservices):

  • Coarse Scaling: Scaling required upgrading or adding entire VMs, which is less cost-effective than granular service scaling.
  • Sequential Failures: In a monolith, a failure in a background task (like a Google Index notification) could occasionally block the main content update thread.
  • Integration Overload: As we integrated with over 20 external news agencies, the "Tight Coupling" of the monolith made it difficult to isolate and debug specific integration failures.
